CloudWatch メトリクス TimeToFirstToken / EstimatedTPMQuotaUsage が追加

2026-03-10

📋 概要

Amazon Bedrock に 2 つの新しい CloudWatch メトリクスが追加されました。TimeToFirstToken はストリーミング API におけるファーストトークンレイテンシーを計測し、EstimatedTPMQuotaUsage は Tokens Per Minute (TPM) クォータの推定消費量を追跡します。

追加設定やオプトイン不要で CloudWatch にそのまま公開され、クライアント側の計測コードなしに推論パフォーマンスとクォータ使用状況をモニタリングできます。

🔄 Before / After

❌ Before
-ファーストトークンレイテンシー計測にはクライアント側の独自実装が必要
-クォータ消費状況をリアルタイムに把握する標準的手段がなく、レート制限に事前対処困難
-SLA ベースライン確立やレイテンシー劣化アラーム設定が容易ではなかった
✅ After
-TimeToFirstToken が自動記録され、クライアント側計測が不要に
-EstimatedTPMQuotaUsage でモデル横断のクォータ消費追跡、プロアクティブなアラーム設定が可能
-追加コストなし、API 変更なし、オプトイン不要で即利用可能

🛠️ 設定方法

前提条件

✓Amazon Bedrock を利用できる AWS アカウント
✓CloudWatch へのアクセス権限 (cloudwatch:GetMetricData, cloudwatch:PutMetricAlarm など)
✓Bedrock の推論 API を使用するアプリケーション

ステップ 1: CloudWatch コンソールでメトリクスを確認

Bash メトリクスの存在確認 
# AWS CLI でメトリクスの存在を確認
aws cloudwatch list-metrics \
  --namespace "AWS/Bedrock" \
  --metric-name "TimeToFirstToken"

オプトインは不要で、推論リクエストを実行すると自動的にメトリクスが記録されます。

ステップ 2: レイテンシー監視アラームの作成

Bash TimeToFirstToken が 5 秒を超えた場合にアラーム発報 
# TimeToFirstToken が 5 秒を超えた場合にアラームを発報する例
aws cloudwatch put-metric-alarm \
  --alarm-name "Bedrock-TTFT-High-Latency" \
  --namespace "AWS/Bedrock" \
  --metric-name "TimeToFirstToken" \
  --statistic "Average" \
  --period 300 \
  --threshold 5000 \
  --comparison-operator "GreaterThanThreshold" \
  --evaluation-periods 2 \
  --alarm-actions "arn:aws:sns:us-east-1:123456789012:bedrock-alerts"

閾値はアプリケーションの SLA に合わせて調整してください。

ステップ 3: クォータ消費量の監視アラームの作成

Bash クォータの 80% 到達でアラーム発報 
# EstimatedTPMQuotaUsage がクォータの 80% に達した場合にアラーム
aws cloudwatch put-metric-alarm \
  --alarm-name "Bedrock-TPM-Quota-Warning" \
  --namespace "AWS/Bedrock" \
  --metric-name "EstimatedTPMQuotaUsage" \
  --statistic "Maximum" \
  --period 60 \
  --threshold 80 \
  --comparison-operator "GreaterThanThreshold" \
  --evaluation-periods 3 \
  --alarm-actions "arn:aws:sns:us-east-1:123456789012:bedrock-alerts"

クォータ消費量が閾値に近づいた段階で事前に通知を受け、レート制限を回避できます。

⚠️ 制限事項・注意点

制限事項

✗TimeToFirstToken はストリーミング API のみ対象。非ストリーミング API では利用不可
✗メトリクスは正常完了リクエストのみ対象。エラーリクエストのレイテンシーは記録されない
✗更新頻度は毎分。秒単位のリアルタイムモニタリングには不向き

考慮すべき点

⚠EstimatedTPMQuotaUsage は「推定」値であり、実際のクォータ消費量とは若干の差異が生じる可能性あり
⚠クォータ上限値はアカウントごとに異なるため、アラーム閾値はアカウントのクォータに合わせて設定が必要

💬 チャットアプリケーションの応答速度監視

生成 AI チャットボットを提供し、最初のレスポンスまでの待ち時間を一定以内に保つ SLA を設定しているシナリオ。

Bash P99 レイテンシーアラーム 
# P99 レイテンシーが 3 秒を超えた場合にアラーム
aws cloudwatch put-metric-alarm \
  --alarm-name "ChatBot-TTFT-P99" \
  --namespace "AWS/Bedrock" \
  --metric-name "TimeToFirstToken" \
  --extended-statistic "p99" \
  --period 300 \
  --threshold 3000 \
  --comparison-operator "GreaterThanThreshold" \
  --evaluation-periods 2 \
  --alarm-actions "arn:aws:sns:us-east-1:123456789012:chatbot-ops"
✅ クライアント側の計測コードなしにレイテンシー SLA を監視でき、劣化時に即座にアラートを受け取れる。
🔀 マルチモデル環境でのクォータ管理

複数の基盤モデルを使い分けるアプリケーションで、モデルごとのクォータ消費量を把握し、上限到達前に増枠を申請するシナリオ。

Bash モデルごとの TPM 消費量を可視化 
# CloudWatch ダッシュボードでモデルごとの TPM 消費量を可視化
aws cloudwatch get-metric-data \
  --metric-data-queries '[
    {
      "Id": "tpm_usage",
      "MetricStat": {
        "Metric": {
          "Namespace": "AWS/Bedrock",
          "MetricName": "EstimatedTPMQuotaUsage"
        },
        "Period": 300,
        "Stat": "Maximum"
      }
    }
  ]' \
  --start-time "2026-03-10T00:00:00Z" \
  --end-time "2026-03-10T23:59:59Z"
✅ モデルごとのクォータ消費傾向を把握し、事前にクォータ増枠をリクエストしてレート制限によるサービス中断を防止。
📊 推論パフォーマンスのベースライン確立

新しい基盤モデルの導入検討時に、ファーストトークンレイテンシーのベースラインデータを収集して比較評価するシナリオ。

Bash 1 週間分の TTFT 統計を取得 
# 1 週間分の TTFT 統計を取得してベースラインを確立
aws cloudwatch get-metric-statistics \
  --namespace "AWS/Bedrock" \
  --metric-name "TimeToFirstToken" \
  --start-time "2026-03-03T00:00:00Z" \
  --end-time "2026-03-10T00:00:00Z" \
  --period 3600 \
  --statistics "Average" "p50" "p99"
✅ モデル間のレイテンシー比較をデータドリブンに行い、アプリケーション要件に最適なモデルを選定。
