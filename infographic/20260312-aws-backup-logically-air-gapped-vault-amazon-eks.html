📋 概要

AWS Backup の論理エアギャップ Vault が Amazon EKS をサポート。論理エアギャップ Vault は、バックアップコピーをデフォルトでロックし、不変として保存する特殊な Vault であり、AWS 所有キーまたはカスタマーマネージドキーによる暗号化で隔離されます。

クロスアカウント・クロスリージョンでの保存、AWS RAM による Vault 共有、マルチパーティ承認によるアクセス制御に対応し、バックアップコピーの事前複製なしに直接リストアを実行できます。

❌ Before
  • EKS クラスターのバックアップを論理エアギャップ Vault に保存できなかった
  • EKS バックアップにイミュータブルかつ論理的に隔離された保護を適用する手段が限定的
  • クロスアカウントでの EKS バックアップ共有と直接リストアのワークフローが複雑
✅ After
  • EKS クラスターのバックアップを論理エアギャップ Vault に直接保存可能
  • バックアップコピーがデフォルトでロック・不変化され、ランサムウェアや不正削除からの保護が強化
  • クロスアカウント・クロスリージョンでの Vault 共有と直接リストアにより、災害復旧が簡素化

🚀 主要機能

🛡️ EKS バックアップの論理エアギャップ Vault 保存

Amazon EKS クラスターのバックアップを論理エアギャップ Vault に保存可能。バックアッププランでプライマリターゲットまたはコピー先として指定でき、同一アカウント、クロスアカウント、クロスリージョンでの保存に対応。

🔒 イミュータブルバックアップコピー

Vault に保存されたバックアップコピーはデフォルトでロック。不変性により、ランサムウェアや悪意ある削除からの保護を実現。AWS 所有キーまたはカスタマーマネージドキーによる暗号化にも対応。

♻️ 直接リストアによる復旧時間短縮

共有された Vault から直接リストアジョブを実行可能。バックアップコピーの事前複製が不要で、AWS RAM またはマルチパーティ承認によるアクセス制御に対応。

⚙ 設定方法

前提条件

✓ AWS Backup が有効化されていること
✓ Amazon EKS クラスターが稼働していること
✓ 論理エアギャップ Vault が作成済みであること
✓ 適切な IAM 権限が設定されていること

Step 1: 論理エアギャップ Vault の作成

Bash 論理エアギャップ Vault 作成 
aws backup create-logically-air-gapped-backup-vault \
    --backup-vault-name my-eks-air-gapped-vault \
    --min-retention-days 7 \
    --max-retention-days 365

Step 2: バックアッププランで EKS を論理エアギャップ Vault に指定

Bash バックアッププラン作成 
aws backup create-backup-plan --backup-plan '{
    "BackupPlanName": "eks-air-gapped-plan",
    "Rules": [
        {
            "RuleName": "daily-eks-backup",
            "TargetBackupVaultName": "my-eks-air-gapped-vault",
            "ScheduleExpression": "cron(0 12 * * ? *)",
            "Lifecycle": {
                "DeleteAfterDays": 90
            }
        }
    ]
}'

Step 3: EKS クラスターをバックアップリソースとして割り当て

Bash バックアップリソース割り当て 
aws backup create-backup-selection \
    --backup-plan-id <plan-id> \
    --backup-selection '{
        "SelectionName": "eks-clusters",
        "IamRoleArn": "arn:aws:iam::<account-id>:role/AWSBackupDefaultServiceRole",
        "Resources": [
            "arn:aws:eks:<region>:<account-id>:cluster/<cluster-name>"
        ]
    }'

技術面

🛡️ ランサムウェア耐性

デフォルトでロックされたイミュータブルコピーにより、悪意ある暗号化や削除からバックアップを保護

⚙ 運用の簡素化

バックアッププランの設定のみで論理エアギャップ Vault への保存が完了し、追加のスクリプトが不要

🔐 マルチパーティ承認

Vault へのアクセスにマルチパーティ承認を適用でき、セキュリティレイヤーを追加可能

⚠ デメリット・制約事項

📏 保持期間の設定必須

論理エアギャップ Vault には最小保持期間と最大保持期間の設定が必須

🔒 変更・削除不可

バックアップコピーは Vault 内で変更や削除ができない (不変性の仕様上)

🔑 CMK 管理が必要

カスタマーマネージドキー使用時は、キーの管理とローテーションを適切に運用する必要あり

💰 コスト考慮

論理エアギャップ Vault のストレージコストは標準 Vault と異なる場合がある。クロスリージョンコピーにはデータ転送コストが発生

💡 ユースケース

🏦 ユースケース 1: ランサムウェア対策としての EKS バックアップ保護

金融機関が EKS 上でマイクロサービスを運用しており、ランサムウェア攻撃によるデータ損失リスクを最小化したい。論理エアギャップ Vault に最小 30 日保持でバックアップを保存。

✅ バックアップコピーがイミュータブルかつ論理的に隔離されるため、攻撃者がバックアップを暗号化・削除することを防止できる。
🌍 ユースケース 2: クロスアカウント災害復旧

マルチアカウント環境で本番 EKS クラスターを運用しており、本番アカウントが侵害された場合でも別アカウントから復旧したい。AWS RAM で論理エアギャップ Vault をターゲットアカウントと共有。

✅ ターゲットアカウントから直接リストアを実行でき、バックアップコピーの事前転送なしに迅速な復旧が可能。
🏥 ユースケース 3: 規制対応のためのイミュータブルバックアップ

医療系 SaaS を EKS 上で運用しており、HIPAA や各種規制でバックアップの不変性と保持期間の厳格な管理が求められている。最小保持期間 365 日で設定。

✅ 最小保持期間の強制とイミュータブル性により、規制で求められるバックアップの保持と改ざん防止を自動的に実現。

📝 まとめ

AWS Backup の論理エアギャップ Vault が Amazon EKS をサポートしたことで、Kubernetes ワークロードに対するイミュータブルかつ論理的に隔離されたバックアップ保護が実現されました。ランサムウェア対策、コンプライアンス要件、災害復旧の強化を検討している EKS ユーザーは、論理エアギャップ Vault の導入を推奨します。

既存のバックアッププランに論理エアギャップ Vault をコピー先として追加することで、段階的な移行も可能です。
