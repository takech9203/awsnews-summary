🪣 Amazon S3 - Account Regional Namespaces

バケット名をアカウントとリージョンにスコープし、グローバル一意性の制約から解放

37
対応 AWS リージョン
$0
追加料金

名前空間サフィックス

📋 概要

Amazon S3 が汎用バケット向けにアカウントリージョナル名前空間を導入しました。この機能により、バケット名を AWS アカウントとリージョンにスコープして作成できるようになり、グローバルに一意なバケット名を探す必要がなくなります。

複数の AWS リージョンにわたって予測可能なバケット名を作成でき、必要な名前がいつでも利用可能であることが保証されます。顧客ごと、チームごと、データセットごとにバケットを使用するワークロードの構築が容易になります。

SCP と IAM ポリシーを使用して、ユーザーがアカウントリージョナル名前空間内でのみバケットを作成するように強制でき、企業全体で一貫したバケット命名規則を適用できます。AWS マネジメントコンソール、S3 REST API、AWS CLI、AWS SDK、AWS CloudFormation から利用可能です。

🔄 Before / After

❌ Before: グローバル名前空間
  • バケット名はパーティション内でグローバルに一意である必要があった
  • 希望するバケット名が他のアカウントで使用中の場合、別の名前を選択する必要があった
  • 複数リージョンで一貫した命名規則の適用が困難
  • バケット削除後に同じ名前を他アカウントが取得できるリスク
  • マルチテナントでの一意な名前生成が複雑
✅ After: アカウントリージョナル名前空間
  • バケット名をアカウント + リージョンにスコープして作成可能
  • 複数リージョンで同じバケット名を使用でき、予測可能な命名が可能
  • SCP / IAM で名前空間の使用を強制可能
  • 名前の所有権がアカウントに紐づき、競合リスクが排除

⚡ 主要機能

🪣 アカウントリージョナル名前空間でのバケット作成
  • バケット名を AWS アカウントとリージョンにスコープして作成可能
  • バケット名の末尾に -an サフィックスを付与
  • 同じバケット名を複数のアカウントおよびリージョンで使用可能
  • CreateBucket API で新しいバケット名前空間リクエストヘッダーを指定して作成
🎯 予測可能なバケット命名
  • 複数リージョンにわたって一貫したバケット名を使用可能
  • 顧客ごと、チームごと、データセットごとに体系的なバケット名を設計可能
  • 希望する名前が常に利用可能であることが保証される
🔒 ガバナンスと命名規則の強制
  • SCP を使用して、組織全体でアカウントリージョナル名前空間の使用を強制可能
  • IAM ポリシーで個々のユーザーやロールに対して名前空間の使用を制御可能
  • 企業全体で一貫したバケット命名規則を適用可能

🔧 技術仕様

名前空間の比較

項目 グローバル名前空間 アカウントリージョナル名前空間
一意性のスコープ パーティション全体 アカウント + リージョン
バケット名の重複 不可 異なるアカウント/リージョンで可能
名前のサフィックス なし -an
他アカウントによる名前取得リスク あり (削除後) なし
複数リージョンで同じ名前 不可 可能

バケット命名規則

項目 詳細
バケット名の長さ 3 文字から 63 文字
使用可能文字 小文字、数字、ピリオド (.)、ハイフン (-)
サフィックス アカウントリージョナル名前空間のバケットは -an で終了
先頭・末尾 英字または数字で開始・終了

🛠️ 設定方法

前提条件

  • AWS CLI が最新バージョンにアップデートされていること
  • 必要な IAM 権限 (s3:CreateBucket) が付与されていること
  • バケットを作成する AWS リージョンが決定していること

ステップ 1: AWS CLI でのバケット作成

Bash アカウントリージョナル名前空間でのバケット作成 
aws s3api create-bucket \
    --bucket my-app-data-an \
    --region ap-northeast-1 \
    --create-bucket-configuration LocationConstraint=ap-northeast-1 \
    --x-amz-bucket-namespace account-regional

--x-amz-bucket-namespace account-regional ヘッダーにより、アカウントリージョナル名前空間での作成を指定しています。

ステップ 2: CloudFormation テンプレート

YAML CloudFormation でのバケット作成 
AWSTemplateFormatVersion: '2010-09-09'
Resources:
  MyBucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketName: my-app-data-an
      BucketNamespace: account-regional

ステップ 3: SCP による名前空間の強制

JSON SCP ポリシー例 
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "RequireAccountRegionalNamespace",
            "Effect": "Deny",
            "Action": "s3:CreateBucket",
            "Resource": "*",
            "Condition": {
                "StringNotEquals": {
                    "s3:x-amz-bucket-namespace": "account-regional"
                }
            }
        }
    ]
}

この SCP を組織のルートまたは OU に適用することで、すべてのメンバーアカウントがアカウントリージョナル名前空間でのみバケットを作成するように強制できます。

⚖️ メリット・デメリット

✅ メリット

ビジネス面

  • グローバルに一意なバケット名を探す必要がなくなり、運用が簡素化
  • 顧客ごとやチームごとに予測可能な命名パターンを使用可能
  • SCP / IAM で企業全体の命名規則を強制でき、コンプライアンス向上

技術面

  • 複数リージョンで同じバケット名を使用でき、IaC テンプレートの再利用性が向上
  • バケット名がアカウントにスコープされ、削除後の名前取得リスクが排除
  • 希望する名前が常に利用可能で、デプロイの失敗リスクが低減

⚠️ 制約・注意点

制限事項

  • バケット名は -an サフィックスで終わる必要がある
  • 既存のグローバル名前空間バケットからの移行は不可 (新規作成が必要)
  • アクセス時にアカウントとリージョン情報が必要

考慮すべき点

  • 既存ツールやスクリプトの更新が必要になる可能性
  • グローバル / アカウントリージョナル名前空間の混在に対するポリシー策定を推奨
  • S3 Transfer Acceleration 使用時はバケット名にピリオド不可 (従来と同様)

💡 ユースケース

🏢 ユースケース 1: マルチテナント SaaS アプリケーション
SaaS プロバイダーが顧客ごとに専用の S3 バケットを作成してデータを分離する必要がある場合。グローバル名前空間では顧客 ID をバケット名に含める際に名前の競合が発生する可能性があった。
Bash 顧客ごとのバケット作成 
# 顧客ごとにバケットを作成
aws s3api create-bucket \
    --bucket customer-12345-data-an \
    --region ap-northeast-1 \
    --create-bucket-configuration LocationConstraint=ap-northeast-1 \
    --x-amz-bucket-namespace account-regional
→ 顧客 ID をそのままバケット名に使用でき、命名の一貫性が保たれ、バケット管理が大幅に簡素化される
🌐 ユースケース 2: マルチリージョンデプロイメント
グローバルに展開するアプリケーションで、各リージョンに同じ構成のバケットをデプロイする必要がある場合。従来はリージョンサフィックスを付けるなどの工夫が必要だった。
YAML 複数リージョンにデプロイ可能な CloudFormation テンプレート 
# CloudFormation テンプレートを複数リージョンにデプロイ
AWSTemplateFormatVersion: '2010-09-09'
Resources:
  AppDataBucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketName: my-global-app-data-an
      BucketNamespace: account-regional
→ 同じ CloudFormation テンプレートを変更なしで複数リージョンにデプロイでき、IaC の再利用性と保守性が向上
🔒 ユースケース 3: エンタープライズガバナンスの統一
大企業の IT 部門が、すべての部門とプロジェクトに対してバケット命名規則を統一し、セキュリティリスクを低減したい場合。
JSON 命名規則を強制する SCP 
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "EnforceNamingConvention",
            "Effect": "Deny",
            "Action": "s3:CreateBucket",
            "Resource": "*",
            "Condition": {
                "StringNotEquals": {
                    "s3:x-amz-bucket-namespace": "account-regional"
                }
            }
        }
    ]
}
→ SCP により組織全体でアカウントリージョナル名前空間の使用を強制し、バケット名の競合リスクを排除すると同時に一貫した命名規則を適用

💰 料金

アカウントリージョナル名前空間は追加料金なしで利用できます。通常の S3 料金が適用されます。

使用量 月額料金 (概算)
バケット作成 無料
ストレージ (S3 Standard、100 GB) 約 $2.30 (東京リージョン)
リクエスト (100,000 PUT/GET) 通常の S3 リクエスト料金

🌍 利用可能リージョン

AWS China および AWS GovCloud (US) リージョンを含む 37 の AWS リージョンで利用可能です。東京リージョン (ap-northeast-1) を含むすべての商用リージョンで使用できます。
