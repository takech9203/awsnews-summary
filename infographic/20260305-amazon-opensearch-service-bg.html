Amazon OpenSearch Service

Capacity Optimized Blue/Green デプロイメント

キャパシティ不足でもデプロイメントを諦めない - バッチ方式による段階的更新
2026 年 3 月 5 日

📝 概要

Amazon OpenSearch Service に Blue/Green デプロイメントの新オプション Capacity Optimized が追加。キャパシティが不足していてもドメイン更新を完了できるようになった。

🔄 Before / After

Before
❌100% キャパシティの事前確保が必須
❌大規模クラスター (30+) でキャパシティ確保困難
❌不足時はデプロイ失敗、リトライ待ち
→
After
✅バッチ方式で段階的にデプロイ可能
✅バッチサイズを自動決定
✅コンソール / API から戦略を選択

⚙️ 2 つのデプロイメント戦略

Full Swap (Default)
📌100% キャパシティを事前確保し一括スワップ
📌従来方式。全ノードを同時に入れ替え
Capacity Optimized (New)
🌟フルスワップを最初に試行
🌟不足時は自動でバッチ方式にフォールバック
🌟バッチサイズはクラスターサイズと利用可能インスタンスから自動計算

📊 戦略比較

項目 Full Swap Capacity Optimized
必要キャパシティ100%可変 (バッチ分)
速度高速 (一括)可変
キャパシティ不足時失敗バッチにフォールバック
推奨サイズ全サイズ30 ノード以上
動作全ノード同時フル試行→バッチ

🔌 API 変更

DeploymentStrategyOptions パラメータが 7 つの API メソッドに追加された。

  • CreateDomain - 作成時に戦略指定
  • UpdateDomainConfig - 既存ドメインの戦略変更
  • DescribeDomain - ドメイン情報に戦略を含む
  • DescribeDomains - 複数ドメイン情報に含む
  • DescribeDomainConfig - 設定に戦略を含む
  • DescribeDryRunProgress - ドライラン状況に含む
  • DeleteDomain - 削除レスポンスに含む
JSON DeploymentStrategyOptions 
{
  "DeploymentStrategyOptions": {
    "DeploymentStrategy": "CapacityOptimized"
  }
}

Default Full Swap 方式 CapacityOptimized Capacity Optimized 方式

🛠️ 設定方法

前提条件: OpenSearch Service ドメインが作成済みで、設定変更権限を持つ IAM ロール/ユーザーがあること。

Step 1: コンソールから設定

OpenSearch Service コンソールでドメインを選択し、Deployment configuration セクションから Capacity Optimized を選択。

Step 2: AWS CLI から設定
Bash 戦略変更 
aws opensearch update-domain-config \
  --domain-name my-domain \
  --deployment-strategy-options \
  '{"DeploymentStrategy": "CapacityOptimized"}'
Step 3: 設定確認
Bash 設定確認 
aws opensearch describe-domain-config \
  --domain-name my-domain \
  --query 'DomainConfig.DeploymentStrategyOptions'

✅ メリット

ビジネス面
📈キャパシティ不足によるデプロイ失敗リスク軽減
⏱️リトライ待機時間の削減でサービスレベル維持
🏢30 ノード以上の大規模クラスター管理効率化
技術面
🔄自動フォールバック (フルスワップ→バッチ)
🤖バッチサイズの自動決定
🔗既存 API との互換性 (パラメータ追加のみ)

⚠️ デメリット・制約

🕐バッチ方式は Full Swap より完了時間が長い場合あり
🔧バッチ更新中は一部ノードが更新途中の状態に
🚫AWS GovCloud では利用不可 (Commercial のみ)
💡30 ノード未満の小規模クラスターは Full Swap 推奨
💡バッチ更新はトラフィックの少ない時間帯に実行推奨

💰 料金

🆓Capacity Optimized 自体に追加料金なし。デプロイ中の追加インスタンスは通常料金が適用。バッチ方式では一度に使用する追加インスタンス数が少ないため、ピーク時の一時コスト軽減の可能性あり。

🌍 利用可能リージョン

全 AWS Commercial Regions

全 OpenSearch / Elasticsearch バージョンで利用可能

💡 ユースケース

1. 大規模ログ分析基盤の更新

50 データノード以上のクラスターでセキュリティパッチ適用が必要だが、リージョン内のインスタンスキャパシティが逼迫。

Bash
aws opensearch update-domain-config \
  --domain-name production-logs \
  --deployment-strategy-options \
  '{"DeploymentStrategy": "CapacityOptimized"}'

効果: キャパシティ不足でもパッチ適用可能

2. マルチテナント SaaS

複数テナント向け大規模クラスターのバージョンアップグレード。フルスワップでは常にキャパシティ確保が課題。

JSON
{
  "DomainName": "saas-search-cluster",
  "DeploymentStrategyOptions": {
    "DeploymentStrategy": "CapacityOptimized"
  },
  "OpenSearchVersion": "2.17"
}

効果: アップグレード遅延リスク軽減

3. ピーク時のクラスター設定変更

EC2 インスタンス需要が高い時期にインスタンスタイプ変更やストレージ拡張が必要。

Bash
aws opensearch update-domain-config \
  --domain-name analytics-cluster \
  --deployment-strategy-options \
  '{"DeploymentStrategy": "CapacityOptimized"}' \
  --cluster-config \
  '{"InstanceType": "r6g.2xlarge.search"}'

効果: ピーク時でもタイムリーにスケーリング

📋 まとめ

Capacity Optimized Blue/Green デプロイメントは、大規模クラスター (30 ノード以上) の運用で特に有用。キャパシティ不足時に自動的にバッチ方式へフォールバックすることで、デプロイメント失敗リスクを大幅に軽減する。大規模 OpenSearch クラスターを運用している場合は、デプロイメント戦略を Capacity Optimized に変更することを推奨。
