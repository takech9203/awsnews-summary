📋 概要

Kiro v0.11 では、エンタープライズチーム向けの MCP レジストリガバナンス、モデルガバナンス、ドキュメント添付 の 3 つの機能が追加されました。セキュリティ・コンプライアンスチームが AI コーディングツールを評価する際に最も求められる MCP サーバー接続の一元管理と AI モデル使用のガバナンスを、IDE と CLI の両方で決定的に適用されるポリシー制御として提供します。

⚡ 主要機能

📋 MCP レジストリガバナンス
  • 管理者が JSON 形式の許可リストファイルを作成し、HTTPS エンドポイントでホスト
  • 管理コンソールでレジストリ URL を設定 → 全クライアントが起動時に取得し 24 時間ごとに再同期
  • レジストリにないローカル MCP サーバーは自動終了され再追加が防止される
  • レジストリで新バージョンが指定されると自動更新・再起動
  • リモートサーバー (streamable-http / sse) とローカルサーバー (npm / PyPI / OCI stdio) の両方をサポート
  • サーバーパラメータは読み取り専用、開発者は環境変数と HTTP ヘッダーのみ追加可能
  • MCP レジストリオープンスタンダードのサブセットに準拠
🤖 モデルガバナンス
  • 管理コンソールの共有設定からモデルの有効/無効を制御
  • 無効化モデルは組織全体の IDE/CLI のモデルセレクターに非表示
  • デフォルトモデルの設定 → 全クライアントに自動適用
  • GA モデル (US/EU リージョナル推論) と実験的モデル (グローバル推論) のステータス確認可能
  • データレジデンシー要件のある組織はグローバル推論の実験的モデルを無効化して対応
📎 ドキュメント添付
  • チャット入力にファイルをペースト/ドラッグ&ドロップで添付
  • 対応形式: PDF, CSV, DOC, DOCX, XLS, XLSX, HTML, TXT, Markdown
  • ネイティブドキュメントブロックとしてモデルに送信、エージェントが読み取り推論
  • 1 メッセージあたり最大 5 ドキュメントまで添付可能
  • テキストや画像と同一プロンプト内で混在利用可能

Step 1: MCP レジストリファイルの作成

JSON MCP レジストリファイル例 
{
  "servers": [
    {
      "name": "my-docs-server",
      "description": "Internal documentation MCP server",
      "version": "1.2.0",
      "remotes": [
        {
          "transport_type": "streamable-http",
          "url": "https://mcp.internal.example.com/docs"
        }
      ]
    },
    {
      "name": "my-local-tool",
      "description": "Local analysis tool from npm",
      "version": "0.5.0",
      "packages": [
        {
          "registry_type": "npm",
          "name": "@myorg/analysis-tool",
          "version": "0.5.0",
          "runtime": "node",
          "transport_type": "stdio"
        }
      ]
    }
  ]
}

Step 2: レジストリファイルのホスティング

Bash Amazon S3 へのアップロード例 
# Amazon S3 にレジストリファイルをアップロード
aws s3 cp mcp-registry.json s3://my-bucket/kiro/mcp-registry.json

Step 3: 管理コンソールでの設定

Kiro 管理コンソールの Settings > Shared settings から以下を設定。

→MCP registry URL: ホストしたレジストリファイルの URL を入力
→Model availability: 有効化して承認モデルのチェックボックスをオン/オフ、デフォルトモデルを選択

設定後、全 Kiro クライアントが次回起動時または 24 時間以内に自動的にポリシーを反映します。

⚠️ 制限事項・考慮点

制限事項

!MCP レジストリガバナンスとモデルガバナンスは Kiro Enterprise ライセンスが必要
!認証は AWS IAM Identity Center, Okta, Microsoft Entra ID のいずれかに限定
!レジストリ同期間隔は 24 時間 (即時反映ではない。ただし起動時には取得)
!モデルガバナンスは既存モデルの無効化のみ。組織独自モデル追加 (BYOM) は非対応

考慮すべき点

*ローカル MCP サーバー実行には適切なパッケージランナー (npx, uvx, docker) が必要
*レジストリファイルのホスティング先の可用性とセキュリティを適切に管理する必要がある
*実験的モデルが GA に変更された際、管理者が手動でモデルを有効化する運用が必要

設定例 モデルガバナンス設定

Claude Sonnet 4.6 (GA, US/EU Regional): 有効
Claude Opus 4.6 (Experimental, Global): 無効

効果: グローバル推論の実験的モデルを無効化し、リージョナル推論の GA モデルのみを提供。データレジデンシー要件に適合しつつ最新 GA モデルを活用。
🔒 UC2: MCP サーバーのセキュリティ管理
シナリオ: 大規模開発組織で数百人の開発者が各自で MCP サーバーに接続、セキュリティチームが接続先を把握・制御できない。
JSON MCP 許可リスト例 
{
  "servers": [
    {
      "name": "approved-docs",
      "description": "社内ドキュメント MCP サーバー",
      "version": "2.0.0",
      "remotes": [{
        "transport_type": "streamable-http",
        "url": "https://mcp.internal.corp.com/docs"
      }]
    },
    {
      "name": "approved-jira",
      "description": "承認済み JIRA 連携サーバー",
      "version": "1.5.0",
      "packages": [{
        "registry_type": "npm",
        "name": "@corp/jira-mcp",
        "version": "1.5.0",
        "runtime": "node",
        "transport_type": "stdio"
      }]
    }
  ]
}
効果: 承認済み MCP サーバーのみを許可リストに登録。開発者は許可リスト外に接続不可。バージョン固定でサプライチェーンリスクも低減。
📄 UC3: ドキュメント分析ワークフロー
シナリオ: 設計書やレビュー資料を Kiro のエージェントに読み取らせ、コードとの整合性チェックや要件確認を行いたい。
実装例 ドラッグ&ドロップで添付 
添付ファイル:
  - requirements.pdf   # 要件定義書
  - api-spec.xlsx      # API 仕様一覧

プロンプト: これらのドキュメントの内容を確認し、
  現在のコードベースとの差異を分析してください
効果: 最大 5 ドキュメントを 1 プロンプトで送信し、エージェントが内容を読み取り分析。手動での要件トレーサビリティ確認を効率化。
