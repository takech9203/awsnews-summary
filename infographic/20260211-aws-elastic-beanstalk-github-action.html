AWS Elastic Beanstalk

GitHub Actions による自動デプロイメント

GitHub リポジトリへのプッシュで Elastic Beanstalk 環境へ自動デプロイ

2026 年 2 月 11 日

概要

AWS Elastic Beanstalk が GitHub Actions をサポートし、GitHub リポジトリへのコードやコンフィグレーション変更のプッシュ時に Web アプリケーションを自動デプロイできるようになりました。GitHub Actions ワークフローで宣言的な設定を使用し、デプロイメントパッケージの作成、S3 へのアップロード、バージョン管理、環境モニタリングなど、デプロイメントライフサイクル全体を包括的に自動化します。

  • Elastic Beanstalk へのデプロイに手動操作や別途 CI/CD ツールの設定が必要

  • GitHub からの継続的デプロイにカスタムスクリプトや外部プラグインが必要

  • GitHub Actions ワークフローファイルに宣言的設定を追加するだけでデプロイ可能

  • IAM と OIDC 認証によるセキュアな連携を実現
  • アプリケーションと環境の自動作成にも対応

ステップ 1: IAM OIDC プロバイダーの設定

Bash OIDC プロバイダーの作成 
aws iam create-open-id-connect-provider \
    --url https://token.actions.githubusercontent.com \
    --client-id-list sts.amazonaws.com \
    --thumbprint-list 6938fd4d98bab03faadb97b34396831e3780aea1

ステップ 2: IAM ロールの信頼ポリシー

JSON IAM ロール信頼ポリシー 
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Federated": "arn:aws:iam::ACCOUNT_ID:oidc-provider/token.actions.githubusercontent.com"
      },
      "Action": "sts:AssumeRoleWithWebIdentity",
      "Condition": {
        "StringEquals": {
          "token.actions.githubusercontent.com:aud": "sts.amazonaws.com"
        },
        "StringLike": {
          "token.actions.githubusercontent.com:sub": "repo:OWNER/REPO:*"
        }
      }
    }
  ]
}

ステップ 3: GitHub Actions ワークフロー

YAML .github/workflows/deploy.yml 
name: Deploy to Elastic Beanstalk

on:
  push:
    branches:
      - main

permissions:
  id-token: write
  contents: read

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::ACCOUNT_ID:role/github-actions-eb-deploy
          aws-region: ap-northeast-1

      - name: Deploy to Elastic Beanstalk
        uses: aws-actions/aws-elasticbeanstalk-deploy@v1
        with:
          application-name: my-application
          environment-name: my-app-env
          region: ap-northeast-1

コードを GitHub リポジトリにプッシュすると、ワークフローが自動的にトリガーされます。GitHub Actions のログと Elastic Beanstalk コンソールでデプロイメントの進捗を確認できます。

メリット

💼 ビジネス面

  • 開発サイクルの短縮: コードプッシュから本番デプロイまで自動化
  • 運用コストの削減: 手動デプロイ作業の排除によるヒューマンエラーの防止
  • 迅速なフィードバック: デプロイメント検証とヘルスモニタリングによる素早い問題検出

🛠️ 技術面

  • セキュアな認証: OIDC による短期トークンベースの認証で長期的なアクセスキー管理が不要
  • 宣言的設定: YAML でデプロイメントの全設定をコードとして管理
  • インテリジェントリトライ: 一時的な障害に対する自動リトライで信頼性を向上

🔧 運用面

  • 完全自動化: デプロイメントパッケージ作成から環境モニタリングまで一貫した自動化
  • バージョン管理: デプロイメントとコードの変更履歴が GitHub で一元管理
  • 環境の自動作成: アプリケーションや環境が存在しない場合の自動作成による初期セットアップの簡素化

制約事項・注意点

制限事項
  • GitHub リポジトリでの使用に限定 -- GitLab や Bitbucket では利用不可
  • GitHub Actions の実行時間制限やランナーの制約に依存
  • OIDC 認証の設定には IAM の理解が必要
考慮すべき点
  • GitHub Actions の無料枠を超えた場合は GitHub 側の利用料金が発生
  • Blue/Green デプロイメントなど複雑な要件には追加の設定が必要
  • 既存の CI/CD パイプラインからの移行には計画的なアプローチが推奨

ユースケース

🚀
Web アプリケーションの継続的デプロイメント
シナリオ: 開発チームが GitHub で管理する Node.js Web アプリケーションを、コードプッシュのたびに自動的に Elastic Beanstalk 環境にデプロイしたい。
実装: aws-elasticbeanstalk-deploy アクションを main ブランチへのプッシュトリガーで設定し、OIDC 認証で AWS に接続。
手動デプロイ作業が排除され、開発者がコードプッシュするだけで自動的にデプロイが完了し、開発サイクルが大幅に短縮
🔀
マルチ環境デプロイメント
シナリオ: ステージング環境と本番環境を持つアプリケーションで、ブランチごとに異なる環境にデプロイしたい。
実装: ワークフローで条件分岐を使用し、develop ブランチでステージング環境、main ブランチで本番環境にデプロイするよう設定。
ブランチ戦略に合わせた自動デプロイメントが実現し、環境間の一貫性が確保
⚡
新規プロジェクトの迅速なセットアップ
シナリオ: 新しい Web アプリケーションプロジェクトの開始時に、Elastic Beanstalk のアプリケーションと環境の作成からデプロイまでを自動化したい。
実装: アプリケーション・環境自動作成機能を活用し、初回のコードプッシュで全リソースを自動作成。
AWS コンソールでの手動操作なしに、GitHub リポジトリの作成とワークフロー設定だけでデプロイ環境が構築

料金

GitHub Action 自体は追加料金なし
以下の関連サービスの料金が適用されます。
項目 料金
Elastic Beanstalk 追加料金なし -- 基盤の AWS リソースに対してのみ課金
Amazon EC2 インスタンスタイプと実行時間に基づく料金
Amazon S3 デプロイメントパッケージのストレージ料金
GitHub Actions パブリックリポジトリは無料、プライベートは月 2,000 分の無料枠あり
