📋 概要

Amazon EKS Auto Mode が、マネージド Kubernetes 機能の拡張ロギングをサポートしました。Auto Mode が管理するコンピュートオートスケーリング (Karpenter)、ブロックストレージ (EBS CSI)、ロードバランシング (AWS Load Balancer Controller)、Pod ネットワーキング (VPC CNI IP Address Management) の各機能を、Amazon CloudWatch Vended Logs のログ配信ソースとして構成できるようになりました。

CloudWatch API または AWS コンソールを使用して設定でき、収集したログは CloudWatch Logs、Amazon S3、Amazon Kinesis Data Firehose に配信できます。CloudWatch Vended Logs は、標準の CloudWatch Logs と比較して低価格で、AWS の組み込み認証・認可によるセキュアなログ配信を提供します。

❌ Before
  • EKS Auto Mode が管理するコンポーネントのログに直接アクセスする方法がなかった
  • マネージドコンポーネントで問題が発生した場合、トラブルシューティングのための可観測性が限定的だった
  • コンピュートオートスケーリングやストレージの動作を詳細に把握することが困難だった
✅ After
  • 各 Auto Mode マネージド機能のログを CloudWatch Vended Logs として収集・配信可能
  • CloudWatch Logs、S3、Kinesis Data Firehose の 3 つの配信先に柔軟にログを送信
  • 標準の CloudWatch Logs より低価格な Vended Logs 料金体系で利用可能
  • 組み込みの AWS 認証・認可によるセキュアなログ配信

EKS Auto Mode の 4 つのマネージドコンポーネントが CloudWatch Vended Logs の配信ソースとして構成され、3 つの配信先にログを送信する構成を示しています。

🚀 主要機能

📋 マネージドコンポーネントログの収集
  • コンピュートオートスケーリング (Karpenter) のログ
  • ブロックストレージ (EBS CSI) のログ
  • ロードバランシング (AWS Load Balancer Controller) のログ
  • Pod ネットワーキング (VPC CNI IP Address Management) のログ
📦 CloudWatch Vended Logs としての配信
  • 各コンポーネントが個別の配信ソースとして構成可能
  • 1 つの配信ソースから複数の配信先にログを送信可能
  • 複数の配信ソースから同じ配信先にログを集約可能
🎯 柔軟な配信先
  • CloudWatch Logs: コンソール、CLI、CloudWatch Logs Insights でクエリ可能
  • Amazon S3: バケットにオブジェクトとして保存、Amazon Athena で分析可能
  • Amazon Kinesis Data Firehose: OpenSearch Service、Redshift などにストリーミング
🔒 組み込みセキュリティ
  • AWS の認証・認可が組み込み
  • クロスアカウントのログ配信にも対応 (PutDeliveryDestinationPolicy API を使用)

⚙️ 設定方法

前提条件:

  • ✓ EKS Auto Mode が有効なクラスターが作成済みであること
  • ✓ CloudWatch Logs API を使用するための IAM 権限があること
  • ✓ 配信先が作成済みであること

ステップ 1: 配信ソースの作成

Bash コンピュートオートスケーリングログの配信ソースを作成 
aws logs put-delivery-source \
    --name "my-eks-auto-compute-logs" \
    --resource-arn "arn:aws:eks:us-west-2:123456789012:cluster/my-cluster" \
    --log-type "AUTO_MODE_COMPUTE_LOGS"

ステップ 2: 配信先の作成

Bash CloudWatch Logs ロググループを配信先として設定 
aws logs put-delivery-destination \
    --name "my-eks-auto-logs-destination" \
    --delivery-destination-configuration \
        '{"destinationResourceArn":"arn:aws:logs:us-west-2:123456789012:log-group:/eks/auto-mode/compute:*"}'

ステップ 3: 配信の作成

Bash 配信ソースと配信先を接続する配信を作成 
aws logs create-delivery \
    --delivery-source-name "my-eks-auto-compute-logs" \
    --delivery-destination-arn "arn:aws:logs:us-west-2:123456789012:delivery-destination:my-eks-auto-logs-destination"

ステップ 4: 他のコンポーネントのログも設定

Bash 残りのマネージドコンポーネントの配信ソースを追加 
# ブロックストレージログの配信ソースを追加
aws logs put-delivery-source \
    --name "my-eks-auto-storage-logs" \
    --resource-arn "arn:aws:eks:us-west-2:123456789012:cluster/my-cluster" \
    --log-type "AUTO_MODE_BLOCK_STORAGE_LOGS"

# ロードバランシングログの配信ソースを追加
aws logs put-delivery-source \
    --name "my-eks-auto-lb-logs" \
    --resource-arn "arn:aws:eks:us-west-2:123456789012:cluster/my-cluster" \
    --log-type "AUTO_MODE_LOAD_BALANCING_LOGS"

# Pod ネットワーキングログの配信ソースを追加

    --name "my-eks-auto-ipam-logs" \
    --resource-arn "arn:aws:eks:us-west-2:123456789012:cluster/my-cluster" \
    --log-type "AUTO_MODE_IPAM_LOGS"

✅ メリット

💼 ビジネス面

  • → 可観測性の向上: Auto Mode が管理するコンポーネントの動作を詳細に把握可能
  • → トラブルシューティングの迅速化: ログを活用して問題の原因を迅速に特定
  • → コスト最適化: Vended Logs の低価格な料金体系により、標準の CloudWatch Logs よりコストを削減

🔧 技術面

  • → 柔軟なログ管理: 4 つのコンポーネントごとに独立したログ配信を設定可能
  • → 複数配信先のサポート: CloudWatch Logs、S3、Kinesis Data Firehose から用途に応じて選択
  • → セキュアな配信: 組み込みの AWS 認証・認可によるログ配信
  • → クロスアカウント対応: 別アカウントへのログ配信も IAM ポリシーで設定可能

⚠️ デメリット・制約事項

🚧 制限事項

  • EKS コントロールプレーンログ (API サーバー、Audit 等) とは別に設定が必要
  • CloudWatch Logs API を使用して設定する必要があり、EKS コンソールからの直接設定ではない
  • ログ配信先に応じた IAM ポリシーやロールの設定が必要

💡 考慮すべき点

  • ログの配信量に応じて CloudWatch Vended Logs、S3、または Kinesis Data Firehose の料金が発生
  • クロスアカウントでログを配信する場合は PutDeliveryDestinationPolicy API で IAM ポリシーの設定が必要
  • 大量のログを出力する環境では、ログの保持期間とストレージコストを考慮した設計が必要

💡 ユースケース

🔎 ユースケース 1: コンピュートオートスケーリングのトラブルシューティング
シナリオ: EKS Auto Mode クラスターで Pod がスケジュールされない問題が発生し、Karpenter のオートスケーリング動作を調査したい。
→ Karpenter のスケーリング判断やノードプロビジョニングのログを確認でき、問題の原因を迅速に特定できる。
📈 ユースケース 2: ストレージとネットワーキングの長期分析
シナリオ: ブロックストレージと Pod ネットワーキングのログを長期保存して、パフォーマンストレンドを分析したい。
→ S3 に低コストで長期保存し、Amazon Athena で過去のトレンドを分析できる。
⚡ ユースケース 3: リアルタイムログモニタリング
シナリオ: ロードバランシングのログをリアルタイムで監視し、異常検知システムと連携したい。
→ ログをリアルタイムで OpenSearch Service にストリーミングし、ダッシュボードやアラートで監視できる。

📝 まとめ

Amazon EKS Auto Mode の拡張ロギング機能により、マネージド Kubernetes コンポーネント (コンピュートオートスケーリング、ブロックストレージ、ロードバランシング、Pod ネットワーキング) のログを CloudWatch Vended Logs として収集・配信できるようになりました。各コンポーネントを個別の配信ソースとして構成でき、CloudWatch Logs、Amazon S3、Amazon Kinesis Data Firehose に柔軟にログを送信できます。Vended Logs の低価格な料金体系と組み込みの AWS 認証・認可により、EKS Auto Mode クラスターの可観測性を効率的かつセキュアに向上させることができます。EKS Auto Mode を利用しているお客様は、この機能を活用してクラスター運用のトラブルシューティングとモニタリングを強化してください。
