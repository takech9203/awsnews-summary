# AWS HealthImaging - JPEG XL サポート

**リリース日**: 2026 年 02 月 02 日
**サービス**: AWS HealthImaging
**機能**: JPEG XL 転送構文サポート

## 概要

AWS HealthImaging が、JPEG XL 転送構文 (1.2.840.10008.1.2.4.112) で非可逆圧縮された医療画像の保存と取得をサポート開始しました。これにより、デジタル病理学のホールスライドイメージングシステムなど、JPEG XL エンコードされた DICOM データを必要とするアプリケーションと HealthImaging を統合することが、これまで以上に簡単になりました。

この機能により、HealthImaging は JPEG XL ロッシー画像データをトランスコーディングせずに保存するため、データの忠実性を維持しながらストレージコストを削減できます。さらに、取得時にトランスコーディングのレイテンシーなしで、JPEG XL 形式で保存された画像フレームを取得できます。

JPEG XL サポートは、AWS HealthImaging が一般提供されている全ての AWS リージョン (米国東部 (バージニア北部)、米国西部 (オレゴン)、アジアパシフィック (シドニー)、ヨーロッパ (アイルランド)) で利用可能です。

**アップデート前の課題**

- JPEG XL エンコードされた医療画像を HealthImaging に保存する際に、トランスコーディングが必要だった
- トランスコーディングによりデータの忠実性が損なわれる可能性があった
- JPEG XL 形式での画像取得時にトランスコーディングのレイテンシーが発生していた
- デジタル病理学のホールスライドイメージングシステムとの統合が複雑だった

**アップデート後の改善**

- JPEG XL ロッシー画像データをトランスコーディングせずに保存可能
- データの忠実性を維持しながらストレージコストを削減
- 取得時にトランスコーディングのレイテンシーなしで JPEG XL 形式で画像フレームを取得可能
- デジタル病理学アプリケーションとの統合が簡素化

## サービスアップデートの詳細

### 主要機能

1. **ネイティブ JPEG XL サポート**
   - JPEG XL 転送構文 (1.2.840.10008.1.2.4.112) をネイティブサポート
   - トランスコーディングなしでデータを保存
   - データの忠実性を完全に維持

2. **低レイテンシー取得**
   - 取得時にトランスコーディング不要
   - JPEG XL 形式で直接画像フレームを取得
   - アプリケーションのパフォーマンスを向上

3. **ストレージコスト削減**
   - 非可逆圧縮による効率的なストレージ
   - トランスコーディングコストの削減
   - 大規模な画像ライブラリのコスト最適化

4. **デジタル病理学統合**
   - ホールスライドイメージングシステムとのシームレスな統合
   - JPEG XL エンコードされた DICOM データを直接処理
   - 既存のワークフローとの互換性

| JPEG XL Lossy | 1.2.840.10008.1.2.4.112 | ✅ 新規サポート |
| JPEG Baseline Lossy 8-bit | 1.2.840.10008.1.2.4.50 | ✅ サポート済み |
| JPEG 2000 Lossy | 1.2.840.10008.1.2.4.91 | ✅ サポート済み |
| JPEG 2000 Lossless | 1.2.840.10008.1.2.4.90 | ✅ サポート済み |
| HTJ2K Lossy | 1.2.840.10008.1.2.4.202 | ✅ サポート済み |

### JPEG XL の特徴

- **高効率圧縮**: JPEG より高い圧縮率を実現
- **高画質**: 非可逆圧縮でも高い画質を維持
- **ロイヤリティフリー**: オープンスタンダードでライセンス料不要
- **デジタル病理学最適化**: ホールスライドイメージングに最適

## 設定方法

### 前提条件

1. AWS アカウントと HealthImaging データストアが作成されていること
2. DICOM データが JPEG XL 転送構文でエンコードされていること
3. AWS CLI または SDK がインストールされていること

### 手順

#### ステップ 1: JPEG XL 画像のインポート

```bash
# AWS CLI を使用して JPEG XL DICOM データをインポート
aws medical-imaging start-dicom-import-job \
    --datastore-id "12345678901234567890123456789012" \
    --input-s3-uri "s3://my-dicom-bucket/jpeg-xl-images/" \
    --output-s3-uri "s3://my-output-bucket/import-results/" \
    --data-access-role-arn "arn:aws:iam::123456789012:role/HealthImagingImportRole"
```

このコマンドは、S3 バケットから JPEG XL エンコードされた DICOM データを HealthImaging データストアにインポートします。

#### ステップ 2: JPEG XL 画像フレームの取得

```bash
# 画像フレームを JPEG XL 形式で取得
aws medical-imaging get-image-frame \
    --datastore-id "12345678901234567890123456789012" \
    --image-set-id "ea92b0d8838c72a3f25d00d13616f87e" \
    --image-frame-information imageFrameId="8765309876543219876543298765432" \
    outfile.jxl
```

このコマンドは、HealthImaging から JPEG XL 形式で画像フレームを取得します。トランスコーディングのレイテンシーなしで直接取得できます。

## メリット

### ビジネス面

- **コスト削減**: トランスコーディング不要によりストレージとコンピューティングコストを削減
- **統合の簡素化**: デジタル病理学アプリケーションとの統合が容易
- **データ忠実性**: 元のデータ品質を完全に維持

### 技術面

- **低レイテンシー**: 取得時にトランスコーディング不要で即座にアクセス可能
- **高効率圧縮**: JPEG より高い圧縮率でストレージを節約
- **標準準拠**: DICOM 標準の JPEG XL 転送構文をサポート

## ユースケース

### ユースケース 1: デジタル病理学のホールスライドイメージング

**シナリオ**: 病理学研究所が大量のホールスライド画像を JPEG XL 形式で保存し、クラウドから高速にアクセスしたい。

**効果**: HealthImaging に JPEG XL 形式で直接保存し、トランスコーディングなしで取得することで、レイテンシーを最小化し、画像の品質を維持できる。

### ユースケース 2: レガシーシステムとの統合

**シナリオ**: 既存のデジタル病理学システムが JPEG XL エンコードされた DICOM データを生成しており、HealthImaging と統合したい。

**効果**: トランスコーディングなしで既存のデータを HealthImaging に移行でき、システム統合を簡素化できる。
