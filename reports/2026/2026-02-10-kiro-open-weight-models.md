# Kiro - オープンウェイトモデルのサポート開始

**リリース日**: 2026 年 2 月 10 日
**サービス**: Kiro
**機能**: オープンウェイトモデル対応 (DeepSeek v3.2, MiniMax 2.1, Qwen3 Coder Next)

## 概要

Kiro IDE および CLI でオープンウェイトモデルが利用可能になった。DeepSeek v3.2、MiniMax 2.1、Qwen3 Coder Next の 3 つのモデルが新たに追加され、ユーザーはタスクの特性に応じて最適なモデルを選択できるようになった。これにより、速度・コスト・品質のバランスを柔軟にコントロールできる。

Kiro は当初からフロンティアモデルを中心に最高品質の AI コーディング体験を提供してきたが、オープンウェイトモデルの急速な進化により、多くの開発タスクでプロプライエタリモデルに匹敵する結果を出せるようになった。今回の対応により、すべてのタスクに最重量モデルを使用する必要がなくなり、ボイラープレート生成やシンプルなリファクタリングなどには高速・低コストのモデルを活用できる。

**アップデート前の課題**

- Kiro で利用可能なモデルは Anthropic Claude シリーズに限定されていた
- 軽量なタスクでもフロンティアモデルを使用する必要があり、クレジット消費が大きかった
- オープンウェイトモデルを使いたいユーザーは別途自分で環境を構築する必要があった

**アップデート後の改善**

- DeepSeek v3.2、MiniMax 2.1、Qwen3 Coder Next の 3 つのオープンウェイトモデルが IDE と CLI の両方で利用可能に
- クレジット乗数が 0.05x〜0.25x と低く、大幅なコスト削減が可能に
- タスクの特性に合わせてモデルを切り替えることで、速度と品質のバランスを最適化できるようになった
- 無料ユーザーを含むすべてのユーザーが利用可能

## サービスアップデートの詳細

### 主要機能

1. **DeepSeek v3.2 (0.25x クレジット乗数)**
   - スパース Mixture-of-Experts アーキテクチャを採用し、タスクごとに必要なパラメータのみをアクティベート
   - エージェンティックワークフローに優れる: マルチステップツールコール、長時間セッションでの状態維持、複雑な推論チェーン
   - 初期コード生成に強いが、複雑なデバッグやコードレビューの品質には課題がある場合がある
   - エージェント構築や込み入ったデバッグセッションに最適

2. **MiniMax 2.1 (0.15x クレジット乗数)**
   - 多言語プログラミングに特化: Rust、Java、Go、C++、Kotlin、TypeScript、JavaScript など幅広い言語をサポート
   - Web、Android、iOS 向けの UI 生成能力が高い
   - フロンティアモデルと比較して複雑なリファクタリングタスクでは課題がある場合がある
   - 多言語での開発やフロントエンド作業が多いチームに最適

3. **Qwen3 Coder Next (0.05x クレジット乗数)**
   - 80B スパース MoE モデルで、トークンあたり 3B パラメータのみをアクティベート
   - コーディングエージェント向けに設計: SWE-Bench Verified で 70% 以上のスコア
   - 256K コンテキストをサポート
   - エラー検出・リカバリ・ツールコールに強い
   - 特に CLI での長時間エージェンティックコーディングセッションに最適
   - 互換性やインテグレーションの一部に慎重な設定が必要な場合がある

## 技術仕様

### モデル比較

| 項目 | DeepSeek v3.2 | MiniMax 2.1 | Qwen3 Coder Next |
|------|--------------|-------------|------------------|
| クレジット乗数 | 0.25x | 0.15x | 0.05x |
| アーキテクチャ | スパース MoE | - | 80B スパース MoE (3B アクティブ) |
| 強み | エージェンティックワークフロー、推論 | 多言語対応、UI 生成 | コーディングエージェント、長コンテキスト |
| コンテキスト長 | - | - | 256K |
| SWE-Bench Verified | - | - | 70%+ |
| 推奨用途 | エージェント構築、デバッグ | 多言語開発、フロントエンド | CLI エージェンティックセッション |

### 利用方法

IDE のモデルセレクターまたは Kiro CLI からモデルを切り替えて使用する。Auto モードとの併用も可能。

## 設定方法

### 前提条件

1. Kiro IDE または Kiro CLI がインストールされていること
2. Google、GitHub、または AWS BuilderID でログイン済みであること

### 手順

#### ステップ 1: Kiro の更新

```bash
# Kiro IDE の場合: アプリを再起動する
# Kiro CLI の場合: 最新版にアップデート
kiro --version
```

最新バージョンの Kiro にアップデートすることで、新しいモデルオプションが利用可能になる。

#### ステップ 2: モデルの選択

```
# IDE の場合: モデルセレクターから選択
# CLI の場合: --model フラグでモデルを指定
kiro --model deepseek-v3.2
kiro --model minimax-2.1
kiro --model qwen3-coder-next
```

タスクの種類に応じて適切なモデルを選択する。

## メリット

### ビジネス面

- **大幅なコスト削減**: クレジット乗数が 0.05x〜0.25x と低く、フロンティアモデル比で最大 95% のコスト削減が可能
- **チーム全体の生産性向上**: 無料ユーザーを含むすべてのユーザーが利用可能で、チーム全体での AI 活用を促進
- **柔軟なリソース配分**: 軽量タスクにはオープンウェイトモデル、複雑なタスクにはフロンティアモデルという使い分けが可能

### 技術面

- **タスク最適化**: タスクの特性に合わせたモデル選択により、速度と品質のバランスを最適化
- **多言語サポート**: MiniMax 2.1 により多数のプログラミング言語での高品質な開発を支援
- **長コンテキスト対応**: Qwen3 Coder Next の 256K コンテキストにより、大規模コードベースでの作業が効率化

## デメリット・制約事項

### 制限事項

- 推論は AWS US East (N. Virginia) リージョンのみで実行される
- 各モデルはフロンティアモデルと比較して特定のタスクで品質が劣る場合がある
- 現在は experimental サポートの位置付け

### 考慮すべき点

- DeepSeek v3.2 は複雑なデバッグやコードレビューで課題が生じる場合がある
- MiniMax 2.1 は複雑なリファクタリングタスクでフロンティアモデルに劣る場合がある
- Qwen3 Coder Next は互換性やインテグレーションで慎重な設定が必要な場合がある
- モデルの特性を理解した上で適切な使い分けが重要

## ユースケース

### ユースケース 1: ボイラープレートコード生成

**シナリオ**: 新規プロジェクトのスキャフォールディングや定型コードの生成

**効果**: Qwen3 Coder Next (0.05x) を使用することで、フロンティアモデル比で 95% のコスト削減を実現しながら、十分な品質のボイラープレートコードを高速に生成可能

### ユースケース 2: 多言語マイクロサービス開発

**シナリオ**: Rust バックエンド + TypeScript フロントエンドなど、複数言語を使用するプロジェクト

**効果**: MiniMax 2.1 の多言語対応と UI 生成能力を活用し、言語を横断した開発を効率化。0.15x のクレジット乗数でコストを抑制

### ユースケース 3: AI エージェント構築

**シナリオ**: マルチステップのツールコールやロングランニングのエージェンティックセッション

**効果**: DeepSeek v3.2 のエージェンティックワークフロー性能を活用し、複雑なエージェント構築を 0.25x のクレジット乗数で実行可能

| モデル | クレジット乗数 |
|--------|--------------|
| DeepSeek v3.2 | 0.25x |
| MiniMax 2.1 | 0.15x |

## まとめ

Kiro にオープンウェイトモデル (DeepSeek v3.2、MiniMax 2.1、Qwen3 Coder Next) が追加されたことで、開発者はタスクの特性に応じてモデルを柔軟に選択できるようになった。特にクレジット乗数が 0.05x〜0.25x と低く設定されており、日常的な開発タスクにおけるコスト効率が大幅に向上する。フロンティアモデルとオープンウェイトモデルを適材適所で使い分けることで、開発ワークフロー全体の最適化を推奨する。
