# Kiro - 新しい Spec タイプ: バグ修正と Design-First 開発

**リリース日**: 2026 年 2 月 18 日
**サービス**: Kiro
**機能**: Bugfix Spec と Design-First Feature Spec

📊 [このアップデートのインフォグラフィックを見る](https://takech9203.github.io/aws-news-summary/20260218-kiro-specs-bugfix-and-design-first.html)

## 概要

Kiro に新しい Spec ワークフローが追加された。Design-First Feature Spec と Bugfix Spec の 2 つの新しい Spec タイプにより、開発者は既存のアプリケーションに対してより構造化されたアプローチでバグ修正や新機能の実装が可能になった。

Design-First Feature Spec は、アーキテクチャを先に設計してから実装する開発ワークフローを提供する。既存のアーキテクチャが明確で、その上に新機能を実装する場合に最適である。Bugfix Spec は、構造化された根本原因分析を通じてバグを修正するワークフローを提供する。

Kiro 0.10 リリースには、Spec ワークフロー以外にも、Hunk ベースの編集による精密なコード変更レビュー、タスクフック、MCP プロンプトのサポートなどの機能が含まれている。

**アップデート前の課題**

- 既存アプリケーションへの機能追加やバグ修正を構造化されたワークフローで行う手段が限られていた
- AI エージェントとの協働でアーキテクチャファーストのアプローチを取ることが困難だった
- バグ修正時の根本原因分析を体系的に行うフレームワークがなかった

**アップデート後の改善**

- Design-First Feature Spec でアーキテクチャファーストの開発が可能になった
- Bugfix Spec で構造化された根本原因分析とバグ修正が可能になった
- Supervised モードで Hunk レベルの変更レビューが可能になった

## サービスアップデートの詳細

### 主要機能

1. **Design-First Feature Spec**
   - アーキテクチャを先に設計してから実装するワークフロー
   - 既存のアプリケーションに新機能を追加する際に最適
   - レイテンシ要件や非同期メッセージングなどのアーキテクチャ決定を事前に定義

2. **Bugfix Spec**
   - 構造化された根本原因分析ワークフロー
   - バグの発生原因を体系的に特定
   - 既存の動作を維持しながら外科的な修正を実施

3. **Hunk ベースの編集**
   - Supervised モードで Hunk レベルの変更レビューが可能
   - 精密な変更制御で意図しない変更を防止
   - コードレビューの粒度を向上

4. **タスクフック**
   - Spec タスクの前後に自動化された作業を実行
   - テスト実行やリンターチェックの自動化
   - 開発ワークフローのカスタマイズ

5. **MCP プロンプトサポート**
   - MCP サーバーがプロンプト、テンプレート、エリシテーションをチャットに直接表示
   - 外部ツールとの統合がさらに強化

## 技術仕様

### Kiro 0.10 の主要変更点

| 機能 | 説明 |
|------|------|
| Design-First Feature Spec | アーキテクチャファーストの機能開発ワークフロー |
| Bugfix Spec | 構造化された根本原因分析によるバグ修正 |
| Hunk ベース編集 | Supervised モードでの精密な変更レビュー |
| タスクフック | Spec タスクの自動化 |
| MCP プロンプト | MCP サーバーからのプロンプト表示 |

## 設定方法

### 前提条件

1. Kiro IDE が最新バージョン (0.10) にアップデートされていること
2. Kiro アカウントにサインイン済みであること

### 手順

#### ステップ 1: Kiro のアップデート

Kiro IDE を最新バージョンにアップデートする。自動更新が有効な場合は自動的にアップデートされる。

#### ステップ 2: Spec タイプの選択

新しいタスクを開始する際に、Design-First Feature Spec または Bugfix Spec を選択してワークフローを開始する。

## メリット

### ビジネス面

- **品質の向上**: 構造化されたワークフローにより、バグ修正や機能追加の品質が向上
- **リスクの軽減**: アーキテクチャファーストのアプローチで設計ミスを早期に発見
- **チーム協働の改善**: Spec ベースの開発でチーム内の共通理解を促進

### 技術面

- **精密な変更制御**: Hunk ベースの編集で意図しない変更を防止
- **自動化の強化**: タスクフックにより開発ワークフローを自動化
- **MCP 統合の深化**: プロンプトサポートで外部ツールとの連携が強化

## デメリット・制約事項

### 制限事項

- 新しい Spec タイプは Kiro 0.10 以降でのみ利用可能
- タスクフックのカスタマイズには設定ファイルの編集が必要

### 考慮すべき点

- 新しい Spec ワークフローへの習熟に時間が必要
- 既存の Spec を新しいタイプに変換する機能は提供されていない

## ユースケース

### ユースケース 1: マイクロサービスへの移行

**シナリオ**: モノリスからマイクロサービスへの移行で、イベント駆動型の非同期メッセージングアーキテクチャを実装

**効果**: Design-First Feature Spec でアーキテクチャを事前に設計し、AI エージェントの支援で効率的に実装

### ユースケース 2: 本番バグの修正

**シナリオ**: 本番環境で発生したバグの根本原因を特定し、既存の動作を維持しながら修正

**効果**: Bugfix Spec の構造化された根本原因分析で、外科的かつ安全なバグ修正を実施

### ユースケース 3: CI/CD 統合

**シナリオ**: Spec タスクの完了時に自動的にテストとリンターを実行

**効果**: タスクフックにより開発フローが自動化され、品質ゲートが組み込まれる

Kiro 0.10 の新しい Spec タイプにより、バグ修正と新機能開発がより構造化されたワークフローで行えるようになった。Design-First Feature Spec はアーキテクチャファーストの開発を支援し、Bugfix Spec は根本原因分析を体系化する。既存アプリケーションに対して AI エージェントと協働で開発を行う際に、これらの新しい Spec タイプの活用を推奨する。
