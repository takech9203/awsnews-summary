# AWS DevOps Agent - インシデント対応と信頼性向上のための AI エージェント (プレビュー)

**リリース日**: 2025 年 12 月 2 日  
**サービス**: AWS DevOps Agent  
**機能**: インシデント調査、根本原因分析、システム改善提案


## 概要

AWS は AWS DevOps Agent のパブリックプレビューを発表しました。これは、インシデント対応、根本原因の特定、過去のインシデントと運用パターンの体系的な分析を通じて将来の問題を防止するフロンティアエージェントです。

フロンティアエージェントは、自律的で大規模にスケーラブルであり、常時介入なしで数時間または数日間動作する新しいクラスの AI エージェントです。AWS DevOps Agent は「常時稼働の自律的なオンコールエンジニア」として機能し、問題が発生すると、メトリクス、ログ、GitHub や GitLab での最近のコードデプロイメントなど、運用ツールチェーン全体のデータを自動的に相関させます。

**アップデート前の課題**

- 本番インシデント発生時、オンコールエンジニアは複数の監視ツールからデータを分析し、最近のデプロイメントを確認し、対応チームを調整する必要があった
- 根本原因の特定に時間がかかり、ステークホルダーへのコミュニケーション管理も負担だった
- サービス復旧後、インシデントの学びを体系的な改善に変換する帯域幅が不足していた

**アップデート後の改善**

- AWS DevOps Agent が自動的にデータを相関させ、根本原因を特定し、ターゲットを絞った緩和策を推奨
- Slack チャネルを使用したインシデント調整とステークホルダーへのリアルタイム更新を自動化
- 過去のインシデントを分析し、将来の問題を防ぐ高影響の改善を特定


## サービスアップデートの詳細

### 主要機能

1. **インテリジェントなアプリケーショントポロジー**
   - システムコンポーネントとその相互作用の包括的なマップを構築
   - デプロイメント履歴を含み、調査中にデプロイメント関連の原因を特定
   - タスク完了に伴い新しいリソースを発見・追加

2. **マルチツール統合**
   - Amazon CloudWatch、Datadog、Dynatrace、New Relic、Splunk との統合
   - GitHub Actions、GitLab CI/CD との統合でデプロイメントを追跡
   - ServiceNow との組み込みサポート、PagerDuty などへの Webhook 対応
   - BYO MCP サーバー機能で Grafana、Prometheus などのカスタムツールを統合

3. **自動インシデント調査**
   - CloudWatch からのメトリクス相関
   - CloudWatch Logs や Splunk などからのログ分析
   - GitHub からの最近のコード変更レビュー
   - AWS X-Ray からのトレース分析

4. **Slack 統合とコミュニケーション**
   - 専用インシデントチャネルでのリアルタイム更新
   - オンコールチームへの通知
   - 調査チャットインターフェースでエージェントと直接対話

5. **改善提案**
   - 即時緩和計画の提供
   - 可観測性、インフラ設定、デプロイメントパイプラインのギャップを特定
   - Kiro などのエージェント開発ツール向けの詳細な実装ガイダンス


## 技術仕様

### サポートされる統合

| カテゴリ | サービス |
|---------|---------|
| 可観測性 | Amazon CloudWatch、Datadog、Dynatrace、New Relic、Splunk |
| CI/CD | GitHub Actions、GitLab CI/CD |
| インシデント管理 | ServiceNow (組み込み)、PagerDuty (Webhook) |
| コミュニケーション | Slack |
| トレーシング | AWS X-Ray |
| カスタム | BYO MCP サーバー (Grafana、Prometheus など) |

### Agent Space の構成

| 項目 | 説明 |
|------|------|
| Agent Space | AWS DevOps Agent がアクセスできる範囲を定義 |
| トポロジー | AWS リソースとその関係を表示 |
| Web アプリ | オペレーター向けの調査インターフェース |
| IAM ロール | AWS リソースへのアクセス権限 |


## 設定方法

### 前提条件

1. AWS アカウント
2. US East (N. Virginia) リージョンへのアクセス
3. 監視対象アプリケーションの AWS リソース

### 手順

#### ステップ 1: Agent Space の作成

AWS Management Console で AWS DevOps Agent セクションに移動し、**Create Agent Space** を選択します。

```
1. Agent Space の名前を入力
2. IAM ロールを作成 (AWS リソースの調査用)
3. Web アプリを有効化 (オプション)
4. Create を選択
```

Agent Space は、単一のアプリケーション、オンコールチーム、または組織全体に合わせて構成できます。

#### ステップ 2: Web アプリアクセスの設定

オペレーターが調査を開始・監視できるよう、Web アプリへのアクセスを設定します。

```
オプション 1: Operator access リンクから直接アクセス
オプション 2: AWS IAM Identity Center でチームアクセスを設定
```

#### ステップ 3: 調査の開始

Web アプリで **Start Investigation** を選択し、以下の情報を入力します。

```
- Investigation details: 調査の詳細
- Investigation starting point: 開始点 (最新アラーム、高 CPU 使用率、エラー率スパイクなど)
- Date and time of the incident: インシデントの日時
- AWS Account ID for the incident: 対象 AWS アカウント ID
```


## メリット

### ビジネス面

- **MTTR の短縮**: 自動的なデータ相関と根本原因特定で平均復旧時間を短縮
- **運用効率**: オンコールエンジニアの負担を軽減し、より戦略的な作業に集中
- **プロアクティブな改善**: インシデントの学びを体系的な改善に変換

### 技術面

- **包括的な可視性**: 複数のツールからのデータを自動的に相関
- **コンテキスト認識**: アプリケーショントポロジーとデプロイメント履歴を理解
- **拡張性**: BYO MCP サーバーでカスタムツールを統合可能


## デメリット・制約事項

### 制限事項

- プレビュー期間中は US East (N. Virginia) リージョンのみで利用可能
- エージェント自体は us-east-1 で実行されるが、任意のリージョンのアプリケーションを監視可能
- プレビュー期間中は月あたりのエージェントタスク時間に制限あり

### 考慮すべき点

- 初期設定には Agent Space の作成と統合の設定が必要
- 効果的な調査には適切な可観測性ツールの統合が重要


## ユースケース

### ユースケース 1: 本番インシデントの自動調査

**シナリオ**: Lambda 関数でエラーが発生し、CloudWatch アラームがトリガーされた

**実装例**:
```
1. ServiceNow からのアラートで自動的に調査開始
2. AWS DevOps Agent がアプリケーションスタックを特定
3. CloudWatch メトリクス、ログ、X-Ray トレースを分析
4. GitHub での最近のコード変更をレビュー
5. 根本原因と緩和策を Slack チャネルに報告
```

**効果**: 手動での調査時間を大幅に短縮し、迅速なサービス復旧を実現

### ユースケース 2: デプロイメント関連の問題特定

**シナリオ**: 新しいデプロイメント後にパフォーマンス低下が発生

**実装例**:
```
1. エラー率スパイクを開始点として調査開始
2. デプロイメント履歴とパフォーマンスメトリクスを相関
3. 問題のあるコード変更を特定
4. ロールバック手順を含む緩和計画を提供
```

**効果**: デプロイメントと問題の因果関係を迅速に特定

### ユースケース 3: 継続的な信頼性改善

**シナリオ**: 過去のインシデントパターンから予防策を導出

**実装例**:
```
1. 過去のインシデントを分析
2. 可観測性、インフラ設定、パイプラインのギャップを特定
3. 優先度付きの改善提案を生成
4. Kiro 向けの実装ガイダンス (spec) を提供
```

**効果**: リアクティブな対応からプロアクティブな改善へ移行
